JBang tries to have decent defaults, for the cases where they do not fit you can use `jbang config` to set up better defaults.

Example, the default editor for `jbang edit` is automatically determined, but sometimes you want it to be one specific editor. Instead of writing `jbang edit --open=code hello.java` to say it is `code` you want open with, you can use `jbang config set edit.open=code`. When you have done that then `jbang edit hello.java` is all that is needed - JBang will now use `code` as the default editor.

== Configure default value for every argument

If you noticed it what JBang configuration does is to map a configuration key to a command line argument. Meaning any command and its command line arguments can be given default values to change the default behavior of jbang. i.e. `edit.open` is the `--open` argument to `edit` command.

Basically any option can be set by taking the full name of the command, which for example is `run` for `jbang run` and `app.list` for `jbang app list`, together with the name of the option without any dashes. So the option `--format` for `jbang app list` becomes `app.list.format`, and `--jdk-providers` for `build` becomes `build.jdkproviders`.

You can see all the available keys by running `jbang config list --show-available` and if you want to see the current values use `jbang config list`.

Additionally, any option can be set by simply using its name, this is especially useful when there are several commands that all have the same option, and you want ot set them all. So if we take the example above where we set the `app.list.format`, we can also simply use `jbang config set format json`. But that will affect _all_ commands that have a format option, like `alias list`, `catalog list`, `jdk list` and others.

In the case that you set both a simple value, like `format`, and a fully named value, like `app.list.format`, the most specific one takes precedence.

== Local vs Global configuration

JBang has built-in default values and any config setting will by default be set on a global scope. Meaning it will apply to any `jbang` run. The default location for this is at `~/.jbang/jbang.properties`.

If you want to have local settings that only apply within a certain directory tree simply go into that directory and run, for example, `jbang config set --file=. edit.open idea`. Then JBang will store that setting in a local `jbang.properties` file which will override any global setting.

To see which keys are configured where and in what order run `jbang config list --show-origin` and you will get something like:

[source,properties]
----
/Users/max/code/personal/jbangdev/jbang/jbang.properties
   edit.open = code
/Users/max/.jbang/jbang.properties
   edit.open = idea
classpath:/jbang.properties
   init.template = hello
   run.debug = 4004
   run.jfr = filename={baseName}.jfr
   wrapper.install.dir = .
----

== HTTP Authentication

When JBang fetches scripts, catalogs, or other resources over HTTPS, it can authenticate using several credential sources. These credentials are applied both to direct HTTP downloads (scripts, catalogs) and to Maven repository access (dependency resolution via `--repos`).

The credential sources are checked in the following order — the first match wins (most specific first):

1. **URL userinfo** — credentials embedded in the URL itself, e.g. `\https://user:pass@host/path`. Sent as Basic auth. (HTTP downloads only — not applicable to `--repos` URLs.)
2. **`.netrc` exact host match** — per-host credentials from `~/.netrc`. Sent as Basic auth by default. Also supports delegating to external credential helpers via `jbang-auth` and changing the HTTP auth scheme via `jbang-auth-scheme` (see <<Credential helpers>> below). An explicit `.netrc` entry for a host takes priority over environment variables.
3. **`GITHUB_TOKEN`** environment variable — used for `github.com`, `*.github.com`, `githubusercontent.com`, and `*.githubusercontent.com` hosts. Sent as `Authorization: Bearer <value>` for HTTP downloads, or as Basic auth for Maven repositories.
4. **`GITLAB_TOKEN`** environment variable — used for `gitlab.com` and `*.gitlab.com` hosts. Sent as `Authorization: Bearer <value>` for HTTP downloads, or as Basic auth for Maven repositories.
5. **`.netrc` `default` entry** — catch-all fallback from `.netrc`, used when no exact host match or environment variable applies.
6. **`JBANG_AUTH_BASIC_USERNAME` / `JBANG_AUTH_BASIC_PASSWORD`** environment variables — global fallback, applied to all hosts. Sent as Basic auth.

For Maven repositories, `~/.m2/settings.xml` `<server>` entries are also honored and take precedence over the sources above when the server `<id>` matches the repository ID.

=== .netrc support

The `.netrc` file is the same credential file used by `curl`, `wget`, `git`, and other tools. If you already have one, JBang picks it up automatically.

On POSIX file systems, JBang ignores `.netrc` if it is readable, writable, or executable by group or others. Use `chmod 600 ~/.netrc`.

Example `~/.netrc`:

```
machine gitlab.mycompany.com
login __token__
password glpat-xxxxxxxxxxxx

machine github.com
login __token__
password ghp_xxxxxxxxxxxx
```

Single-line entries are also supported:

```
machine gitlab.mycompany.com login __token__ password glpat-xxxxxxxxxxxx
```

A `default` entry acts as a fallback for any host without a specific match:

```
default
login anonymous
password user@example.com
```

Passwords containing spaces can be double-quoted using wget/curl-style quoting:

```
machine example.com
login myuser
password "my secret password"
```

Backslash escapes work inside quoted strings: `\"` for a literal double-quote, `\\` for a literal backslash.

=== Custom .netrc file path

By default JBang reads `~/.netrc`. To use a different file:

```bash
# Via command-line option
jbang --netrc /path/to/my-netrc run script.java

# Disable .netrc lookup for one run
jbang --no-netrc run script.java

# Via jbang.properties (applies to all runs)
jbang config set netrc /path/to/my-netrc
```

=== GitLab token

For `gitlab.com` specifically, you can set the `GITLAB_TOKEN` environment variable as a quick alternative to `.netrc`:

```bash
export GITLAB_TOKEN=glpat-xxxxxxxxxxxx
```

This sends `Authorization: Bearer <token>` for all requests to `gitlab.com` and its subdomains.

NOTE: For self-hosted GitLab instances, use `.netrc` instead — `GITLAB_TOKEN` only applies to `gitlab.com`.

=== Credential helpers

If you already have credentials stored in your OS keychain, the GitHub CLI, or the GitLab CLI, JBang can use them directly via the `jbang-auth` key in `.netrc`:

```
machine github.com
login myuser
jbang-auth gh-auth
```

The value is a comma-separated list of auth methods, tried in order. Available methods:

* **`gh-auth`** — runs `gh auth token --hostname <host>` to get a token from the GitHub CLI. Works for any host where `gh auth login` has been run. Uses `login` from the `.netrc` entry as the username if present.
* **`glab-auth`** — runs `glab config get token --host <host>` to get a token from the GitLab CLI. Works for any host where `glab auth login --hostname <host>` has been run. Uses `login` from the `.netrc` entry as the username if present, otherwise `__token__`.
* **`env.PASSWORD`** — reads a token/password from the `PASSWORD` environment variable. The name is uppercased, so `env.github_token` reads `GITHUB_TOKEN`. Uses `login` from the `.netrc` entry as the username if present.
* **`env.USERNAME-PASSWORD`** — reads both username and password from environment variables, for example `env.github_user-github_token` reads `GITHUB_USER` and `GITHUB_TOKEN`.
* **`git-credential`** — runs `git credential fill` to obtain credentials from whatever credential backend git is configured to use (macOS Keychain, Windows Credential Manager, GNOME Keyring, credential-store, etc.). JBang sets `GIT_TERMINAL_PROMPT=0` so git will never prompt for a username/password interactively — if no stored credential is found, it fails immediately and JBang moves on to the next method.

By default `.netrc` credentials are sent using Basic auth for HTTP downloads. To use another HTTP header style, add `jbang-auth-scheme`:

```
machine gitlab.mycompany.com
jbang-auth glab-auth
jbang-auth-scheme bearer
```

Supported values are `basic` (default) and `bearer`. This only affects direct HTTP downloads; Maven repository authentication still uses Maven username/password credentials.

You can chain multiple methods — the first one that returns credentials wins:

```
machine gitlab.mycompany.com
jbang-auth env.gitlab_token,glab-auth,git-credential
```

You can also combine `jbang-auth` with regular `login`/`password` as a final fallback — if all auth methods fail, JBang falls back to the `login`/`password` values:

```
machine ci.example.com
login deploy
password s3cret
jbang-auth glab-auth,git-credential
```

When the credential helper needs to look up a different host than the one in the `machine` line (e.g., `maven.pkg.github.com` needs credentials from `github.com`), use `jbang-auth-host` to specify the lookup host:

```
machine maven.pkg.github.com
login maxandersen
jbang-auth gh-auth
jbang-auth-host github.com
```

This calls `gh auth token --hostname github.com` and uses the resulting token for requests to `maven.pkg.github.com`.

Credential results are cached per-host for the lifetime of the JBang process, so the subprocess is only invoked once per host.

NOTE: `jbang-auth`, `jbang-auth-host`, and `jbang-auth-scheme` are JBang-specific extensions to the `.netrc` format. They are silently ignored by `curl`, `wget`, `git`, and other tools that read `.netrc`.

=== Maven repository authentication

Credentials from `.netrc`, environment variables, and git credential helpers are automatically applied to Maven repositories specified via `--repos` or `//REPOS` directives. For example:

```bash
# Credentials for maven.pkg.github.com will be looked up from
# GITHUB_TOKEN, .netrc, or jbang git-credential automatically
jbang --repos "ghp=https://maven.pkg.github.com/myorg/myrepo/" \
     com.example:my-artifact:1.0.0
```

If you also have `~/.m2/settings.xml` with a `<server>` entry whose `<id>` matches the repository ID (e.g. `ghp` in the example above), the `settings.xml` credentials take precedence.

==== Example: GitHub Packages without secrets in .netrc

Using `gh-auth` with `jbang-auth-host`, you can access GitHub Packages Maven repositories without storing any tokens or passwords in `.netrc`. The GitHub CLI (`gh`) manages the credentials for you:

```
machine maven.pkg.github.com
jbang-auth gh-auth
jbang-auth-host github.com
```

Then resolve artifacts directly:

```bash
gh auth refresh -s read:packages
jbang --repos "myrepo=https://maven.pkg.github.com/myorg/myrepo/" \
     com.example:my-artifact:1.0.0
```

JBang calls `gh auth token --hostname github.com`, uses the returned token as the password, and authenticates to `maven.pkg.github.com`.

NOTE: GitHub Packages Maven repositories require authentication even for public packages. Make sure the GitHub token has package read access:
+
```bash
gh auth status
gh auth refresh -s read:packages
```

This also works for GitHub Enterprise Server instances — just point `jbang-auth-host` to your GHES hostname and add `login` with your enterprise username:

```
machine maven.pkg.github.mycompany.com
login myuser
jbang-auth gh-auth
jbang-auth-host github.mycompany.com
```

As long as `gh auth login --hostname github.mycompany.com` has been run, JBang will use the stored token.

=== Debugging authentication

Use `--verbose=network` to see which credential source JBang selected for each request without unrelated
diagnostic output:

```bash
jbang --verbose=network run https://gitlab.mycompany.com/group/project/-/raw/main/script.java
```

The output will include lines like `Using .netrc credentials for host: gitlab.mycompany.com` or `Using GITHUB_TOKEN environment variable for host: github.com`.

== Proxy configuration

If you use proxies there are a few settings to apply to make JBang *fully* work with proxies. Unfortunately Java treats proxy setup a little bit different based on OS and environment.


First the proxy settings for bash/zsh scripting part (needed if jbang nor java is not already installed). If you use Windows, OSX or Gnome Linux and have proxy configured it *should* just but if not setting the `http_proxy` and `https_proxy` environment variables will enable JBang scripts to access the proxy. For example:

```bash
http_proxy=http://proxyhost:8888
https_proxy=http://proxyhost:8888
```

Second the proxy settings for Java itself. This is done by setting the `JAVA_TOOL_OPTIONS` environment variable. For systems with proxy setup built in you can use system proxies like this:

```bash
JAVA_TOOL_OPTIONS="-Djava.net.useSystemProxies=true"
```

if that does not work set it using Java http.proxy properties:

```bash
export JAVA_TOOL_OPTIONS="-Dhttp.proxyHost=proxyhost -Dhttp.proxyPort=8888 -Dhttps.proxyHost=proxyhost -Dhttps.proxyPort=8888"
```

Finally JBang uses maven resolver to download dependencies. Maven resolver uses the `settings.xml` file to configure proxies. The `settings.xml` file is usually located in `~/.m2/settings.xml`. For example:

```xml
<settings xmlns="http://maven.apache.org/SETTINGS/1.0.0"
          xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
          xsi:schemaLocation="http://maven.apache.org/SETTINGS/1.0.0
                              http://maven.apache.org/xsd/settings-1.0.0.xsd">
    <proxies>
        <!-- For HTTP -->
        <proxy>
            <id>http-proxy</id>
            <active>true</active>
            <protocol>http</protocol>
            <host>proxyhost</host>
            <port>8888</port>
        </proxy>
        <!-- For HTTPS -->
        <proxy>
            <id>https-proxy</id>
            <active>true</active>
            <protocol>https</protocol>
            <host>proxyhost</host>
            <port>8888</port>
        </proxy>
    </proxies>
</settings>
```
